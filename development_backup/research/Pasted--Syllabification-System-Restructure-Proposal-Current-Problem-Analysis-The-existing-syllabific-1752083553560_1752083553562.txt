
# Syllabification System Restructure Proposal

## Current Problem Analysis

The existing syllabification system in `server/utils/syllable.ts` has become overly complex with multiple patches and workarounds. Key issues:

1. **Fragmented Logic**: Rules scattered across different functions with unclear precedence
2. **Band-aid Fixes**: Multiple override systems trying to fix fundamental flaws
3. **Poor Error Patterns**: Words like "experience" → "exp•eri•enc•e" show systematic failures
4. **Maintenance Burden**: Each new word error requires another override rather than fixing root causes

## Proposed New Architecture

### 1. Clean Foundation-First Approach

**Phase 1: Core Syllable Detection**
- Build robust vowel sound detection using CMU phoneme data
- Ensure each syllable has exactly one vowel sound
- Handle vowel clusters correctly (ai, ea, oo, etc.)

**Phase 2: Educational Rule Hierarchy**
```
Priority 1: Morphological Boundaries (educationally critical)
Priority 2: Phonetic Patterns (pronunciation accuracy)  
Priority 3: Structural Rules (open/closed syllables)
```

### 2. Simplified Rule Structure

**Morphological Processor**
- Identify prefixes (un-, re-, pre-, dis-, etc.)
- Identify suffixes (-ing, -ed, -tion, -ly, etc.)
- Preserve root word integrity
- Handle compound words

**Phonetic Processor** 
- Apply consonant cluster rules consistently
- Handle special letter combinations (ch, sh, th, etc.)
- Manage silent letters and irregular patterns

**Fallback Processor**
- Apply basic CV (consonant-vowel) patterns
- Default consonant placement rules
- Open syllable preferences

### 3. Implementation Strategy

**New File Structure:**
```
server/utils/syllable/
├── core.ts           // Main CMUSyllabifier class
├── morphological.ts  // Prefix/suffix detection
├── phonetic.ts      // Sound-based rules
├── patterns.ts      // Common pattern definitions
└── fallback.ts      // Basic CV rules
```

**Data-Driven Approach:**
- Replace hardcoded overrides with pattern recognition
- Use CMU phoneme data for sound-based decisions
- Implement rule testing framework

### 4. Expected Benefits

1. **Systematic Accuracy**: Fix entire word classes, not individual words
2. **Maintainability**: Clear rule hierarchy and separation of concerns
3. **Educational Value**: Proper morphological awareness
4. **Extensibility**: Easy to add new patterns and rules

### 5. Testing Strategy

- Comprehensive test suite with educational word lists
- Regression testing against current working words
- Performance benchmarking
- Grade-level appropriate vocabulary validation
